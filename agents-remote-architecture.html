Architecture Guide

Agents Remote 架构全景

两个仓库、三层服务、四条通信通道

STATUS · Living doc DATE · 2026-05-27
TL;DR
agents-remote-core 的 server 在 PTY 中运行 Claude/Codex CLI，解析终端输出 + 注入 hooks 获取权威状态，全量快照写入共享内存（.mq）。agents-remote 的 lark 客户端通过 WebSocket 连接飞书，每秒轮询共享内存渲染卡片。用户交互通过飞书回调 → lark handler → Unix socket → server → PTY/hook response 闭环。两个仓库通过共享内存文件解耦，进程完全独立。

01两个仓库的定位

agents-remote-core
PTY Server + Hook Harness
管理 Claude/Codex CLI 进程生命周期。负责 PTY 创建、终端输出解析（pyte）、hook 注入和状态追踪、共享内存写入。不关心飞书。
agents-remote
Lark Client + 快捷 CLI
飞书机器人（WebSocket 长连接）、卡片渲染、用户交互处理。同时提供 cla/cl/cx 等快捷命令。不管 PTY。
当前状态
agents-remote 内部也有一份 server/server.py（历史遗留），cla/cl 快捷命令启动的是这个旧 server。要用到 agents-remote-core 的 hook 能力，需要改为启动 core 的 server，或将 core 的 hooks.py 移植过来。

06Claude vs Codex 差异

维度Claude CodeCodex CLI
Hook 注入--settings '{"hooks":...}' 内联 JSON合并到 ~/.codex/hooks.json（文件发现）
PermissionRequest通过 Notification + PermissionRequest 事件独立 PermissionRequest 事件
AskUserQuestionPreToolUse 双向拦截 → updatedInput.answers不存在（Plan Mode 内建交互）
终端解析器claude_parser.py（分割线 + 圆点 + 星号）codex_parser.py（背景色区域 + › 提示符）
Hook 清理无需（--settings 是进程级参数）cleanup 时按标记从 hooks.json 移除注入项

07启动流程

方式 A：agents-remote CLI（当前）

cla / cl / cx / cdx
  → agent_remote.py cmd_start()
    → tmux 新会话，运行 agents-remote/server/server.py（旧 server，无 hooks）
    → agents-remote lark start（后台守护进程，WebSocket 连飞书）

方式 B：agents-remote-core server（需切换）

agents-remote-core start <session>
  → tmux 新会话，运行 agents-remote-core/server/server.py
    → HookHarness 创建 FIFO + 脚本
    → Claude/Codex CLI 带 --settings / hooks.json 启动
    → HookState 写入 .mq 共享内存
  → agents-remote lark start（读同一个 .mq，自动适配 hook_state 字段）
要点
两种方式的 lark 客户端完全相同——它只读 .mq，不关心是哪个 server 写的。差别仅在 server 端是否注入了 hooks。方式 B 下 hook_state 字段有值，lark 客户端自动走 hook 模式；方式 A 下该字段为 None，自动降级到终端解析模式。

Key paths

  • /tmp/agents-remote/<hash>.mq — 共享内存文件
  • /tmp/agents-remote/<hash>.sock — Unix socket
  • /tmp/agents-remote/<hash>_hooks/ — FIFO + 脚本 + 响应文件
  • ~/.agents-remote/ — 日志、绑定、配置
  • ~/.codex/hooks.json — Codex hook 配置（动态注入/清理）
